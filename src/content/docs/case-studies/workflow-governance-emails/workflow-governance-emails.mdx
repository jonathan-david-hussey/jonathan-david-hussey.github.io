## Context

The R&D governance portal exists to carry ordered approval decisions. A project proposal is raised by an initiator and moves through a configured chain of approvers, and at every stage the platform has to direct attention to the one person expected to act next. Approval, rejection, request for change, update after change, completion, and the finance handoff are communication events as much as they are workflow events.

The assets that keep this activity trustworthy are therefore broader than the email body. The workflow instance, its activity stream, the current approver, the initiator, the approver chain, the email templates, the rendered subject and HTML body, the recipient list, the SMTP configuration, the correlation id, and the notification audit record all contribute to business continuity. If any one of these assets is incomplete or untraceable, the organisation can no longer prove that a governance action reached the person who was expected to act on it.

That reframing is what makes notification reliability a governance concern rather than a cosmetic messaging feature. The design had to guarantee that emails were selected from the same state model as the workflow, sent outside the user request path, and preserved as evidence whenever they mattered.

## The Problem

The central threat was silent communication failure during approval governance. If an approver is never notified, a workflow stalls. If the wrong recipient is notified, confidentiality and accountability are both affected. If an email is sent but no record is kept, the platform cannot say whether a delay was caused by the system, the recipient, or an upstream workflow issue.

Two common implementation choices are themselves vulnerabilities. Sending SMTP inline from the request that updates workflow state can fail halfway, leaving a changed database row and no reliable notification history behind it. A hard-coded template switch is a second weakness: business-owned wording and routing rules become entangled with deployment cycles, so a copy change waits on a release.

The risk analysis pointed to one conclusion. A visible, incomplete notification can be diagnosed and replayed; an apparently successful send that leaves no evidence cannot be trusted at all. The system therefore needed a notification model in which every communication could be traced back to a persisted workflow activity, rendered from configured templates, dispatched through durable infrastructure, and recorded after a real production send.

## Constraints

- **Ordered approval stages.** The current approver changes as the workflow advances, so a notification has to follow the active stage rather than a static recipient list.
- **Request-for-change loops.** A request for change moves responsibility to the initiator and then back to the same approver after updates, without walking the workflow backward through the approver chain.
- **Template governance.** Seven workflow email outcomes had to be mapped through data records rather than embedded as controller-level conditionals.
- **Asynchronous failure.** Rendering, recipient lookup, SMTP delivery, and persistence can each fail independently after the user's action has already returned.
- **Replay and diagnosis.** Operators needed a safe way to replay a stage after data correction or a transient failure, using the same handlers as normal production flow.
- **Non-production safety.** Lower environments had to exercise template rendering and SMTP dispatch without emailing real executives or recording misleading production-style notification proof.

## The Design

The design can be read as a safeguard model around detection, protection, and recovery, built on one durable boundary: the workflow activity stream. That stream records user actions, transitions, notifications, and errors, and the notification model treats it as the evidence from which email routing is derived. Detection comes from those persisted activities and the notification records written after a real send; protection comes from history-led selection, safe Fluid rendering, message validation, and the durable Wolverine boundary that keeps sending out of the request path; recovery comes from a single normalised error route and guarded replay endpoints that reuse the production handlers.

The user-facing workflow command stays deliberately thin. Approval, rejection, and request-for-change actions update the immediate workflow facts and publish events into Wolverine. The slower, more failure-prone concerns (deciding the next transition, selecting notification types, rendering email, and talking to SMTP) happen behind durable message handling, where a fault can be retried or surfaced rather than lost.

Template selection is history-led by design. The template selector loads the workflow instance together with its workflow definition, its email template mappings, and its activity stream. It then finds the most recent action activity and matches that action's workflow status and activity status against the email template mapping. The selected child template types are the notification outcomes that should be built: an approval, a completion, or a finance handoff on project creation, for example. Because the notification is derived from the persisted activity that caused it, a stage can be regenerated long after the original request state has disappeared. This is the property that makes replay safe.

The email builder acts as the dispatcher from a selected template type to the builder that owns it. The supported outcomes are created, rejected, approved, request for change, request for change updated, completed, and a project-created finance handoff. Dedicated builders keep recipient and model construction close to each workflow scenario, while the shared email message carries the common output: workflow instance id, To, Cc, Bcc, subject, body, and child template type.

Rendering is isolated behind a dedicated Fluid rendering service. It parses the parent and child template strings with Fluid, registers only the known workflow context models, and adds a small set of controlled custom filters, such as one that joins approver names into a readable list. That gives business-owned templates flexibility without granting a template arbitrary code execution. The rendering surface is a safeguard, not an open door.

The send path is the next control boundary. The notification handler receives a notification event, builds the messages for the workflow instance, and publishes a send-email event for each one. The send handler then validates that the message exists, carries at least one recipient, has a subject, and has a body. An invalid message produces a workflow error event instead of a malformed outbound email, so a bad payload is prevented from reaching the mail server at all.

SMTP delivery is handled by a dedicated SMTP sender through MailKit: it builds a MIME message, connects with StartTLS, authenticates, sends the HTML body, and disconnects. This is also where non-production safety is enforced. If a test address is configured, the recipients are replaced by the test mailbox, so staging can execute the identical path without reaching a real approver. If no test address is configured, the send is treated as production: the handler persists a notification record containing the serialized email payload and writes a notification activity row for each To and Cc recipient. The audit evidence is created by the real send, not promised by the test one.

Request-for-change handling is the most sensitive route, because it must not lose the workflow's current stage simply because an approver needs corrections. The handler reads the latest request-for-change action activity: if the approver requested changes, it creates pending transition activities for the initiator; if the initiator has updated the request, it returns the task to the current approver. The stage stays fixed while responsibility moves according to persisted activity state.

Errors follow a single recovery route. Workflow handlers emit a workflow error event carrying the workflow instance id, correlation context, error text, and stack trace. A dedicated error handler marks the workflow instance with an error state, writes a system-owned error activity, and emits a normalised application error event with metadata. A notification failure therefore stays inside the workflow evidence model and is handed to the platform's broader error handling rather than vanishing into a log line.

The result is a notification architecture shaped around governance risk. It names the assets that make an email accountable, derives routing from durable history, validates every outbound message, keeps SMTP away from the workflow command, redirects lower-environment sends safely, and records production notification evidence where an operator can query it.
