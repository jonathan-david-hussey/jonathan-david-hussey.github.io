## Context

The production AI platform supports agents whose core activity extends beyond producing text. Voice and chat conversations can require an external action, so the platform needs a dependable way to expose those capabilities and carry each request to the appropriate service.

Both channels need access to the same kinds of tools, but they do not submit calls or consume results in the same form. The integration layer keeps the underlying capability consistent while allowing each channel to retain the contract expected by its runtime.

The intended outcome is a configurable boundary between agents and external services. New tools can then be introduced without placing service contracts, credentials, validation rules, or response transformations inside every agent.

## The Problem

The central difficulty was reconciling variable model-generated arguments with deterministic external API contracts. A model can select the correct tool while still producing names or values that do not match what a service expects.

The variation continues beyond the request: services apply different authentication schemes, naming conventions, payload structures, response sizes, and error formats; voice and chat also require different result shapes. A request can also be structurally correct while failing an application-specific property check, which means that execution cannot rely on schema validation alone.

The problem was therefore one of containment. The agent contract had to remain stable while the execution path absorbed external variation, applied configured security and validation rules, returned useful failure context, and left an observable outcome for subsequent processing.

## Constraints

- **Multiple Channels.** The system was designed to make voice and chat converge on shared orchestration while preserving the result form required by each channel.
- **Dynamic Capabilities.** The available tools had to vary according to the tenant, agent, and configured task rather than appear as one fixed catalogue.
- **Credential Isolation.** Authentication had to remain outside prompts, generated tool arguments, and published events.
- **Configurable Security.** The platform had to apply property-level validation strategies before an external action while keeping those rules outside agent and adapter code.
- **Prompt-Controlled Guidance.** Prompt engineers had to be able to configure the instruction returned when a request failed a rule, allowing error context to remain appropriate to the application.
- **Cross-Application Reuse.** The same endpoint had to support different channels and applications without forcing each caller to inherit identical request policies.
- **Contract Variability.** The integration layer had to absorb differences in arguments, naming styles, payloads, and response structures.
- **Context Economy.** The platform was designed to return only the response fields and collection sizes needed for the current task.
- **Extensibility.** A new external service had to be a bounded extension through configuration, routing, and a focused adapter rather than a rewrite of the orchestrator.
- **Operational Continuity.** In the event of either success or failure, the exchange had to produce a consistent event boundary for follow-up processing.

## The Design

The design uses one progressive pipeline: channel-specific extraction, a canonical request, runtime configuration, request normalisation, policy validation, bounded routing, response filtering, channel adaptation, and event publication. Each stage contains one kind of variation before handing a smaller and more predictable contract to the next stage.

Voice and chat entry points first pass through channel-aware extractors. These components remove the source envelope and produce a canonical tool request, allowing the remaining stages to operate without branching around the original channel.

Tool discovery is handled before execution. Data-backed definitions describe the permitted endpoints, arguments, types, required fields, and task-specific guidance; the platform uses those definitions to generate the tools available to an agent. Runtime configuration then resolves the service location and authentication that apply to the tenant and agent, keeping connection details out of the generated request.

Once the request has a complete runtime context, normalisation absorbs differences in argument naming and shape. The canonical request can also carry a declarative policy written through a DSL-style interface. Its configuration identifies the evidence to inspect, the property-level strategy to apply, the comparison input, and the guidance to return when a rule does not pass.

The policy is evaluated before the target action. Supporting information can be obtained through the same routing boundary when a rule needs it, after which the selected strategies determine whether execution can continue. A failed rule stops the action and returns its configured message through the normal channel response, giving the agent useful context without embedding the decision in its prompt.

This configuration allows prompt engineers to consume endpoints through different platforms and applications while retaining control over the permitted behaviour. Each caller can apply its own validation rules, error context, and task-specific instructions without creating a separate endpoint or changing the shared orchestrator.

Once validation succeeds, a routing strategy selects the service boundary, and a focused adapter contains the implementation required by that external contract. The orchestrator therefore coordinates the sequence without accumulating every service-specific rule.

Responses return through the same path in reverse. Optional filters can select nested fields and limit large collections before channel adapters produce a serialised voice result or a structured chat result. The model receives the information required for the current task rather than an unrestricted service response.

Every exchange also reaches a separate event boundary. Authentication is removed before the request and result are published through Redis, allowing asynchronous listeners to respond without receiving connection details or becoming part of the original tool call.

A new integration is therefore introduced through a configured contract, optional request policies, a routing choice, and a focused adapter. The result is one execution language for voice and chat: the layer discovers permitted tools, resolves runtime context, validates application-specific conditions, contains service variation, shapes the response, and records a sanitised outcome without moving those responsibilities into agent logic.
