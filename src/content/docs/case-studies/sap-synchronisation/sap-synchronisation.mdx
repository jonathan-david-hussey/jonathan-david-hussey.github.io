## Context

Finance produced periodic reports by gathering data from different departments and sources. The figures were brought into spreadsheets, combined by hand, and shaped through formulas and pivot tables until they formed the report required for the financial period.

SAP already held the financial records that should drive this work. The portal provided an opportunity to consume those records directly and generate a report comparable to the spreadsheet Finance had been maintaining.

The exchange also had to work in reverse. The portal managed project approval, while SAP needed the resulting project information for financial processing. The integration therefore connects one reporting flow into the portal with a separate project flow back to SAP.

## The Problem

The spreadsheet process placed the work of integration inside the report itself. Finance had to collect files, reconcile different inputs, maintain formulas, rebuild pivot tables, and repeat the same preparation for each reporting period before the figures could be reviewed.

Automating that work required more than copying rows from SAP. Several related datasets contributed to the financial picture, and the portal had to preserve their meaning well enough to produce the expected report. An interrupted refresh also needed to remain visible rather than appearing as a complete result.

Project information introduced a different ownership rule. Financial data moved from SAP into the portal, but approved project information moved from the portal into SAP. The design had to make both directions explicit while keeping their responsibilities understandable.

## Constraints

- **Financial Authority.** The system was designed to use SAP as the source for financial data rather than continuing to consolidate departmental spreadsheets.
- **Report Continuity.** The portal had to generate a financial report comparable to the spreadsheet output Finance already used.
- **Two Directions.** The system had to ingest financial datasets from SAP while reconciling approved project information back to SAP.
- **Integration Protocol.** The clients were designed to handle SAP's paged OData interface and the authenticated requests required by both directions.
- **Safe Refreshes.** The inbound path had to avoid presenting incomplete source data as a successful refresh.
- **Recoverable Processing.** In the event of an interrupted run, the integration had to retain enough progress information to identify the affected data group.

## The Design

The integration begins with the reporting outcome. SAP supplies the financial data, the inbound path brings it into the portal, and the portal produces the report Finance previously prepared in Excel. Project approval forms a second path, allowing the relevant information to be reconciled back to SAP.

A manual request and a nightly Hangfire schedule provide the two entry points. Both begin the same shared refresh, ensuring that operator-initiated and scheduled processing follow one route. Hangfire determines when scheduled work begins; the integration logic remains outside the scheduler.

The shared refresh prepares tracked work for each data group before processing starts. Wolverine then coordinates the durable execution path, allowing the work to continue independently while remaining connected to the original refresh.

The inbound datasets share one retrieval pattern. OData provides a consistent interface for reading data from SAP, while the .NET service transforms the responses and stores the information needed by the portal. This common path keeps the integration repeatable without exposing the business structure of each dataset.

Imported data is connected to portal projects through supporting relationships, including orders where required. These relationships give the reporting layer the context needed to present financial information against the appropriate projects without making the report responsible for integration work.

The refresh process keeps incomplete source responses separate from accepted reporting data. This means that an upstream interruption remains an integration issue rather than silently becoming the basis of a Finance report.

Project reconciliation follows the other direction. Approved project information is compared with what SAP currently holds, and the necessary changes are sent through the outbound path. The same capability can participate in the shared refresh or respond when an approval is completed.

The remaining technology has deliberately narrow responsibilities. Hangfire owns scheduling, Wolverine provides durable execution, SQL Server supports application and processing state, and the SAP clients contain the OData communication boundary. Together, these parts allow the integration to record incomplete work without exposing internal business structures in the public design.

The result is a reporting flow rather than a catalogue of domain entities. SAP remains the source of the financial data, the portal recreates the report Finance previously assembled manually, approved project information travels back to SAP, and the shared refresh provides enough evidence to identify work that requires attention.
