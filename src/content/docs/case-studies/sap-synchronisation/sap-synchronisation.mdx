## Context

The R&D governance portal is the place where project requests, approvals, and operational governance are carried out. SAP is the place where financial and organisational reference data is maintained. The system therefore contains two classes of assets that have to remain in agreement: SAP-owned finance data flowing into the portal, and portal-owned project records flowing back to SAP.

The important assets were not only the records themselves. Orders, Accounts, Centres, Financial figures, Employee figures, project codes, project status, SAP payloads, workflow completion events, and audit evidence all form part of the operational system. If any of these assets becomes stale, corrupted, or untraceable, the result is a business-continuity problem rather than a narrow technical defect.

Before this work, agreement between the systems depended on manual intervention. Finance reference data could become stale in the portal, and a project that had passed through governance still required manual re-entry before SAP reflected it. The objective was to implement a synchronisation model that could be triggered both routinely and deliberately, and that would leave enough evidence behind to understand each run after the fact.

## The Problem

The central threat was data drift between the governance portal and SAP. This manifested in several ways: reference data could be old, project records could be absent from SAP after approval, SAP failures could disappear into application logs, and retry behaviour could make it difficult to determine whether a failed run had stopped permanently or was still being recovered.

The problem was therefore not simply "integrate with SAP." The system needed to protect the integrity and availability of finance and project data while making synchronisation failures diagnosable. A successful design had to determine what was sent, what SAP returned, which entity failed, whether the message would be retried, and whether the final status represented completion, recoverable error, or permanent failure.

The risk analysis led to one practical conclusion: silent failure was more dangerous than visible incompletion. A partial run that clearly identifies the failed entity can be recovered; an apparently successful process that leaves no audit trail cannot be trusted.

## Constraints

- **Two triggers, one behaviour.** The nightly refresh and the manual refresh button had to enter the same execution path so that operational and user-initiated runs could not diverge.
- **Bidirectional responsibility.** Six domains flow from SAP into the portal, while projects flow from the portal into SAP after governance approval.
- **SAP OData v2 behaviour.** Clients had to manage JSON-formatted responses, server-reported total counts, page-size and offset paging, optional date filters, mutating requests, and CSRF token acquisition.
- **Durable execution.** A process crash or transient SAP failure should not force a human to guess which entity completed and which one needs retrying.
- **Auditable evidence.** The platform needed entity-level status, attempts, request JSON, response data, correlation id, and error details.
- **Safe local persistence.** Inbound feeds had to refresh and upsert data without using a bad SAP response as a reason to mass-delete local finance data.
- **Extensibility.** New SAP entities had to be added through a consistent pattern rather than through a fresh integration design each time.

## The Design

The design can be understood as a safeguard model around detection, protection, and recovery. Detection comes from explicit ledger and log records. Protection comes from durable events, SAP client boundaries, retry policies, and upsert-only inbound persistence. Recovery comes from status-aware handlers that distinguish queued work, in-progress work, retryable error, permanent failure, and completion.

A global refresh has two entry points. A sync API endpoint handles the manual refresh button, while a Hangfire recurring job covers the nightly run. Both entry points publish the same global refresh event, so the behaviour after triggering is identical. Hangfire remains a scheduler and dashboard; it does not contain the SAP workflow.

The global refresh handler then establishes the audit boundary. It creates one ledger row per entity (Financial, Accounts, Orders, Centres, Employees, and Projects), all under a single correlation id. Each row begins as pending with an initial log entry. Only after that evidence exists does the handler emit a start refresh event.

Each entity has a small start handler derived from the same base class. The start handler finds the pending ledger row for its entity, records that the entity has entered the queued stage, and emits the entity-specific process event. The process handler then marks the row as in progress, stores the serialized refresh request, records how many delivery attempts the message has taken, calls the refresh service, and emits a refresh completed event when the entity succeeds.

The inbound refreshes are deliberately uniform. A shared SAP client builds OData v2 requests with JSON formatting, server-reported total counts, page size, skip values, and entity-specific filters where needed. A shared refresh service reads the first page, calculates total pages, then continues paging through SAP responses with a default batch size of 100. Repositories map SAP response models into local entities and upsert by entity-specific keys. For example, Financial records use a composite identifier, while other finance entities use their own stable match keys.

This is the key safeguard for incoming data. The local database is refreshed by update and insert, not by deletion. If SAP returns an empty or partial response, the result is a visible refresh outcome rather than a destructive local purge.

The SAP client boundary also contains integration-specific controls. A delegating handler adds basic authentication to every request, sets the appropriate accept headers, and fetches a CSRF token from SAP metadata before any mutating create or update call. That keeps a repeated OData concern in one place instead of duplicating it across entity clients.

Projects follow the opposite direction. The project reconciliation service loads the current SAP project list, loads local portal projects with valid project identifiers, maps local overall statuses into SAP active or inactive states, and then reconciles the two sides. Missing SAP projects are created; changed project names or statuses are updated. The SAP project client also trims project names to the SAP length boundary before create or update calls.

Workflow completion is treated as an operational trigger. When the workflow completion handler validates a completed workflow instance, it emits a project sync event alongside the workflow notification event. That means a newly approved project can be reconciled to SAP immediately, while the same project entity also remains part of the wider global refresh model.

Durability is provided by Wolverine rather than by hand-rolled background code. The application persists its messages in SQL Server, uses durable inbox and outbox policies, applies Entity Framework Core transactions, and keeps local queues for start, completion, workflow email, and application error events sequential where ordering matters. Retry behaviour uses configured Wolverine retry counts and jittered cooldowns; the process handler records whether an exception is still a recoverable error or has become a final failed state after the retry limit.

The result is a synchronisation layer with a clear risk response. The system identifies the assets that can drift, creates audit records before work starts, isolates SAP mechanics behind shared clients, uses a repeatable pattern for inbound reference data, handles project reconciliation as a special outbound case, and turns every run into a recoverable sequence of statuses rather than an opaque scheduled task.
