## Context

An internal R&D governance platform needed to agree with SAP about money and people: internal orders, GL accounts, profit centres, cost centres, utilization and manpower figures all live in SAP, while project records originate in the portal. Before this work, that agreement was manual. Finance data went stale, and newly approved projects had to be keyed into SAP by hand.

## The Problem

Replace the manual process with a synchronisation layer that finance staff can trust and engineers can extend ‚Äî where a failed sync is visible, diagnosable and recoverable, not silent.

## Constraints

- **Two triggers, one behaviour.** Finance wanted a nightly refresh, but users needed an on-demand button ‚Äî and both had to run exactly the same logic.
- **Auditable by non-engineers.** A failed sync couldn't just vanish into a log file; someone had to be able to answer "did last night's refresh run, what did it send, and what came back?"
- **SAP's OData v2 quirks.** Paged responses, CSRF tokens on mutating calls, and no natural keys on several entities.
- **Asymmetry.** Six domains flow *from* SAP into the portal; projects flow *out* of the portal into SAP ‚Äî one pipeline had to carry both directions.

## The Design

One pipeline, two triggers. A Hangfire recurring job fires nightly and a `POST /sync/refresh` endpoint serves the button ‚Äî both do nothing except publish the same `GlobalRefreshEvent`. Hangfire stays a scheduler and operator dashboard; it knows nothing about SAP. All actual work happens in Wolverine's durable, SQL-backed message pipeline.

The fan-out point creates one ledger row per entity, each seeded `Pending`. Per-entity start handlers mark their row `Queued` and emit a process event; process handlers mark it `InProgress`, capture the serialized request, call the refresh service, and finish by recording either a completion (with the full response serialized alongside) or an error. Failure classification is retry-aware: the handler reads Wolverine's envelope attempt count and distinguishes "errored, will retry" from "failed, gave up".

### Inbound: Six Feeds, One Pattern

Every inbound entity follows the same four-part pattern ‚Äî an OData client, a `ToEntity()` mapping extension, a bulk-upsert repository, and a paging refresh service. The base client handles OData v2 mechanics (`$format`, `$inlinecount`, `$top`/`$skip`); the base service pages through results a hundred at a time using the server-reported count; the base repository bulk-upserts with per-entity match keys ‚Äî order number for internal orders, account-plus-company-code for GL accounts, hashed composite IDs where SAP has no natural key (utilization rows hash eight fields).

Cross-cutting concerns live in one place. A delegating handler on every SAP HTTP client deals with basic auth, content negotiation and ‚Äî the part nobody wants to write twice ‚Äî CSRF token acquisition for mutating calls. Resilience (exponential retry with jitter) is applied at client registration, not sprinkled through business code.

### Outbound: Projects Flow the Other Way

Projects invert the pattern. The portal is the source of truth ‚Äî projects are born there through a governed approval workflow ‚Äî so the sync *reconciles SAP against the portal*: load the current SAP project list, compare against locally approved projects, `POST` the missing ones, `PUT` the changed ones.

There are two ways into this path. The nightly global refresh includes projects like any other entity. But when a project's approval workflow completes, the workflow's completion handler also emits an immediate sync event ‚Äî so a newly approved project appears in SAP within moments of the final signature, without waiting for the night run.
