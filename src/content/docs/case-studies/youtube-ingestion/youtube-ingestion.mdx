*Senior Software Engineer · 2019 - 2024*

**At a glance**

- 70k+ channels refreshed through 50-channel Data API batches
- Each channel eligible for feed refresh every 48 hours
- Serverless fan-out across SQS, S3, Kinesis, DynamoDB and Elasticsearch

## The situation

A social-media intelligence platform needed YouTube data to behave like a first-class source: recent videos searchable alongside other content, channel profile details kept current, and stale or deleted accounts removed from the ingestion loop.

The awkward part was that YouTube was really two different ingestion problems. RSS feeds were the right source for recent videos: cheap, simple, and available per channel. The YouTube Data API was the right source for account metadata: channel title, description, thumbnails, reach and media counts. Treating those as one pipeline would either waste quota or under-refresh the data that mattered.

So the system split them deliberately: one continuous RSS pipeline for video content, and one quota-aware Data API pipeline for channel metadata.

## The feed pipeline

The RSS side starts with a scheduler that selects channels due for refresh. Each channel is eligible after a 48-hour window, and the picker Lambda runs every minute, loading a configurable batch of channel IDs from SQL Server and pushing them to SQS.

From there the work is broken into small, recoverable stages:

```text
scheduled channel picker
  -> SQS channel queue
  -> fetch feed Lambda
  -> API Gateway controlled XML fetch
  -> S3 Processing object with metadata
  -> S3 event / SQS notification
  -> parse feed Lambda
  -> header update stream + video upload queue + feed-state queue
  -> DynamoDB post store
  -> DynamoDB stream
  -> Elasticsearch bulk index
```

The API Gateway hop is not decorative. It gives the team a throttle point in front of YouTube's XML feed endpoint, so if the provider starts pushing back, the system has a knob to turn without rewriting the Lambdas.

The fetch stage hashes the feed before doing more work. If the feed has not changed, it exits early. If it has changed, it writes the XML to S3 under a `Processing` prefix and includes the correlation ID and serialized handoff DTO in object metadata. That handoff is small but important: the parse Lambda can recover all context from the object event and the object itself, without relying on process memory or a side channel.

## Turning feeds into product data

Parsing fans the XML in two directions. Header changes go to a Kinesis stream so account names and descriptions can be updated separately. Video entries go to SQS in batches, where another Lambda converts each entry into a DynamoDB record.

DynamoDB is doing more than acting as a staging table. It stores compressed source XML, carries query keys for video ID and month-based recovery, applies a ten-year TTL, and emits a stream. The stream processor turns those records into Elasticsearch documents: title, description, URL, thumbnail, views, likes, extracted URLs, extracted hashtags, and normalized index text.

That gives the system a useful operational shape. S3 is the feed-level audit and retry boundary; DynamoDB is the post-level recovery store; Elasticsearch is the search surface. Each has a narrow job instead of one database pretending to be all three.

Feed state is explicit too. Successfully parsed XML is moved from `Processing` to `Success`; failed XML goes to `Failure`. Success objects expire quickly, failure objects are retained longer for debugging. That sounds mundane, but it is the kind of detail that makes an ingestion system operable when thousands of tiny jobs are moving at once.

## The Data API pipeline

The account-metadata path runs on a different cadence. A daily scheduler reads active YouTube accounts from SQL Server, excluding channels already known to be inactive, and queues account records for refresh.

The next Lambda consumes up to 50 channel IDs at a time because that is the natural shape of the YouTube `channels/list` API. At the scale in the source system, 70k+ channels translated to roughly 1,450 API calls per day against a 10,000-request daily quota. That batching choice turned quota from a production constraint into a capacity margin.

Transient failures are retried with exponential backoff and jitter. API responses that represent permanent business conditions are handled differently: deleted or missing channels can be marked inactive, and downstream account updates only proceed when channel data exists. The final stage updates the platform account record with the current title, description, image, subscriber reach and media counts, then schedules the affected profile cache to refresh.

## Designed for failure

The architecture assumes individual channels, messages and provider calls will fail. Each queue has a dead-letter queue. Dead-letter queues have CloudWatch alarms. Lambdas have deliberate batch sizes and concurrency limits. Correlation IDs are carried through SQS message attributes and S3 metadata so a single channel refresh can be followed across the system.

The code also keeps replay paths simple. A feed can be manually reintroduced through the S3 `Processing` prefix. A channel can be targeted directly through the queue. DynamoDB retains the compressed source entry long enough to rebuild a search document if the indexing side needs attention. Monthly Elasticsearch indexes are trimmed by a scheduled housekeeping Lambda rather than left to grow without a policy.

## What this demonstrates

The interesting work here was not "call YouTube and save the result." It was separating two data sources with different economics, then building the pipeline so the cheap path could run continuously, the quotaed path could batch intelligently, and failures stayed local.

That is the pattern I would reuse: make the unit of work small, give every handoff a durable boundary, preserve enough source data to recover, and spend API quota only where it buys fresher product data.
