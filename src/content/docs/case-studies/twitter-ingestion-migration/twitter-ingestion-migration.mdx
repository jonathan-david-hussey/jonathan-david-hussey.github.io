## Context

The platform depended on Twitter data for search, reporting and influencer profiles. That meant the ingestion system had to keep hundreds of thousands of accounts fresh, pull new posts without wasting API calls, backfill important gaps, and preserve compatibility with downstream systems already built around an older tweet JSON shape.

## The Problem

The hard part was not the initial download. It was keeping the whole loop healthy under provider limits: deciding who really needed a refresh, paginating until the useful range was exhausted, prioritizing urgent backfills without starving normal refreshes, and migrating to Twitter v2 without forcing every downstream consumer to change at once. A long-standing production defect in the pagination path — silently capping ingestion yield at 6.25% per request — made clear that pagination had to be treated as a first-class design concern, not plumbing.

## Constraints

- **Provider rate limits** — every wasted call against an account with no new data was a call not spent on one that had some.
- **A frozen downstream contract.** Search indexing, reporting and enrichment all consumed the older v1-style tweet document; the migration could not force them to change together.
- **Backfill urgency without starvation.** Recovering missing history was usually more urgent than routine refresh, but it must never permanently crowd refresh out.
- **Search API ceilings** on the status-update loop, independent of Twitter's own limits.

## The Design

The system is a set of cooperating loops, not a single batch job.

```text
user refresh scheduler
  -> candidate queue
  -> refresh validator
  -> tweet ingester queue
  -> ECS tweet ingester
  -> Kinesis tweet stream
  -> last-tweet/status update Lambdas

backfill scheduler
  -> priority queue
  -> same ECS tweet ingester

ingestion-status scheduler
  -> status queue
  -> Search API status updater
  -> SQL status tables
```

The refresh scheduler reads the oldest eligible accounts from SQL Server and pushes candidate IDs to SQS in batches of ten. The validator then asks a cheaper question before scheduling expensive work: has the account's tweet count or most-recent tweet changed since the last check, and is the account still accessible? Only then does it create a media download request. That small gate keeps the download workers focused on accounts with new data rather than cycling through every account on every pass.

### Pagination as a First-Class Concern

The tweet ingester is an ECS task that scales from zero when either the normal queue or priority queue has work. Its processor turns a download request into a bounded time window: if the request has no explicit IDs, fake tweet IDs are generated from timestamps, normally covering the searchable retention window. If the caller accidentally reverses the oldest and newest IDs, the processor swaps them before calling the API.

After each download, the parser calculates the minimum and maximum tweet IDs returned. If tweets were found and the minimum ID is still newer than the requested oldest ID, the ingester requeues the same user with a lower max ID. That continuation rule is the guardrail:

```text
download window
  -> parse min/max tweet IDs
  -> push tweets to Kinesis
  -> if more history remains, requeue with max = min - 1
```

The reason this was treated so carefully was hard-won: the yield-capping defect lived exactly here, and fixing this path restored the intended yield from 6.25% to 100%.

### Priority Without Starvation

Backfill candidates are tracked in their own state table. A scheduled Lambda prepares missing backfill records, evaluates which users need work, and pushes download requests to a priority queue. The same ECS ingester consumes both queues, using a configurable preference for the priority queue: by default, it attempts the priority queue three times before returning to the normal queue. If the priority queue is empty, it immediately falls back to normal refresh.

### Migrating to v2 Without Breaking Consumers

Twitter v2 changed the shape of tweet data — `data`, `includes`, expansions, referenced tweets, media, polls, places and errors spread across related collections. Downstream systems still expected the older v1-style status document, so the parser package became the compatibility layer: it splits a v2 response into per-tweet documents, attaches only the relevant includes, and converts each into the older status model with v2-only fields preserved under an extension object.

One subtle problem was missing referenced tweets. A quote or retweet can point at a tweet that is not present in the original includes collection. The ingester detects those references, performs a lookup for the missing IDs, merges the result back into the response, and only then runs conversion. If a response cannot be safely converted, that user window is skipped rather than poisoning the stream.

### Operating the Pipeline

SQS queues have dead-letter queues and age alarms. The ECS services scale out when queues contain messages and scale back to zero when they drain. The queue scheduler understands rate-limit exceptions and waits until the provider reset time before resuming. Kinesis writes are batched, failed records are retried, and the last-tweet updater stores the newest processed tweet ID per user so the next refresh has a clean starting point.

The ingestion status loop runs separately because it depends on the search surface. A scheduler can enqueue hundreds of thousands of Twitter IDs, using TPL Dataflow to parallelize SQS batch sends. The updater deliberately processes one ID at a time per Lambda invocation and caps concurrency because the Search API has its own ceiling.
