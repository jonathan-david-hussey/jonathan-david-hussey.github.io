## Context

A social-media intelligence platform needed email reports that behaved differently depending on the client's intent. Real-time reports were operational: "tell me when matching content appears." Program-insights reports were analytical: "summarize how a program changed over the week."

## The Problem

Two very different reporting promises had to share one operable backend. Real-time reports needed short, repeatable windows, tolerance for ingestion delay, and a clean skip path when no posts existed. Program insights needed weekly current-versus-previous-period comparisons, topic trends, most-engaged influencers, and a richer template — with validation that fails a bad program cleanly instead of sending a broken email. Both needed durable artifacts, traceable sends and post-send feedback.

## Constraints

- **Ingestion delay** — posts can arrive minutes after the event they belong to; an alerting window that ignores this silently loses content.
- **Search API ceilings** — expensive analytical queries had to be bounded so reporting never degraded the platform.
- **Alert fatigue** — an empty "nothing happened" email trains clients to ignore the product.
- **Auditable sends** — support had to be able to answer exactly what was sent, to whom, and whether it arrived.
- **Non-production safety** — the same pipeline had to be testable in staging without emailing real clients.

## The Design

The core pipeline is a queue-backed chain of small stages that branches by report type and converges for delivery.

```text
real-time report scheduler
program-insights scheduler
  -> report processor queue
  -> report processor Lambda
  -> generate-data queue / real-time generate-data queue
  -> generate-data Lambda
  -> S3 JSON artifact
  -> email queue
  -> send-email Lambda
  -> SES bulk templated email

SES callbacks
  -> SNS
  -> callback queue -> status updates + Elasticsearch email logs
  -> ChurnZero queue -> customer-success activity tracking

preview notification
  -> preview processor
  -> report processor queue
```

The report processor creates a run request, attaches the `RunId` to the message, and then branches on report type. Real-time schedules are routed to a dedicated real-time generate-data queue; program-insights schedules use the standard one. Both generate-data event sources process one message at a time and cap concurrency at two, which keeps expensive Search API work bounded and makes failures easy to isolate by schedule run.

### Real-Time Reports

Real-time scheduling is deliberately windowed. The scheduler rounds the current UTC time down to the nearest 15-minute interval, then shifts the query window back by a minimum 15-minute ingestion buffer. In practice, a scheduler tick around `12:06` normalizes to `12:00`, searches `11:30` to `11:45`, and leaves the most recent 15 minutes alone so late-arriving tweets can still be captured on the next run.

The real-time generate-data path fetches the selected topics and influencers, builds account maps for the configured channels, asks the Search API for matching posts, prepares post HTML, and records influencer activity. If there are no posts and the request is not a preview, the send-email stage skips the real-time email entirely.

### Program Insights

The weekly scheduler checks the configured day and hour in the schedule's timezone and prevents another run until at least seven days have passed. When a schedule is due, it builds a reporting window from the previous Saturday at midnight through Friday at `23:59:59.999`, then compares current and previous periods across program overview, trending topics, most-engaged influencers and per-influencer topic engagement.

Before doing the full request, the generator validates topic queries with a small Search API call so a bad topic shape fails the schedule cleanly instead of producing a broken template. Search calls are wrapped with a Polly retry proxy that distinguishes transient failures (retry with backoff) from expected data-generation exceptions (exit immediately) — "try again later" and "this program is not valid for insights" need different operational behaviour.

### Sending Without Losing the Audit Trail

Both report paths produce a durable artifact before any email is sent: the generate-data Lambda writes the report JSON to S3 under a path containing the schedule ID and run ID, then queues the send-email stage. The send-email Lambda reads that artifact, asks SES to render the template, saves the rendered HTML back to S3 as `RenderedTemplate-{runId}.html`, then sends a bulk templated email tagged with schedule ID, preview status and environment. Per-recipient message IDs and errors are written back to the database so callbacks can be reconciled to the original run. An override email setting reroutes all recipients in staging-like environments.

### Closing the Loop

SES delivery, bounce and complaint events arrive through SNS and are processed from queues. The callback processor updates request status and bulk-indexes email-log documents into a month-based Elasticsearch index. A separate ChurnZero processor consumes the same topic through its own queue and forwards engagement events to customer success — post-send tracking never couples to the send path. Previews exercise the same machinery end to end: the preview processor marks a schedule as preview, applies preview window rules, and feeds the normal report processor queue.
