## Context

The platform is a social-media intelligence service whose clients expect email reports to behave according to their intent. Some reports are operational, an alert that says in effect "tell me when matching content appears". Others are analytical, a weekly summary that says "show me how a program changed over the week". One backend had to keep both promises without confusing the two.

The following assets have been identified as central to that activity: the two reporting promises themselves, the schedules that decide when each report is due, the finite search budget the reports draw upon, the generated report data, the rendered emails, the recipients, the record of what was sent, and the delivery and engagement feedback that follows. We are able to regard these as assets because client trust rests upon them; any negative impact, whether a missed alert, a broken summary, or a send that cannot be accounted for, would affect not one email but the client's confidence in the product.

This framing turns reporting into a question of which promises must be protected, and by which safeguards, rather than a single task of querying data and sending mail.

## The Problem

The central difficulty was that two very different promises had to share one operable backend, and the threats to each were different. A real-time alert needs short, repeatable windows and a tolerance for the fact that posts arrive minutes after the event they belong to; if the window ignores that ingestion delay, content is silently lost. A weekly summary needs current-versus-previous comparisons, trending topics, and the most-engaged influencers, drawn from expensive analytical queries that must be bounded so they never degrade the platform for everyone else. Both also share a subtler threat: an empty "nothing happened" alert trains a client to ignore the product, and a send that leaves no trace cannot be explained when the client asks whether it arrived.

The level of risk followed from the likelihood of each threat combined with the severity of its impact. The most damaging outcomes were the silent ones: a real-time report that quietly drops late-arriving posts, or a weekly report that sends a broken template, each erodes trust without presenting any visible failure, and heavy weekly analytics blocking a time-sensitive alert would be as damaging as either. Our analysis therefore favoured a design that isolated the two promises where they differ, bounded the expensive work, made every send auditable, and treated an empty alert as something to be suppressed rather than sent.

## Constraints

- **Ingestion delay.** Posts can arrive minutes after the event they belong to, so an alerting window that ignores this silently loses content.
- **A finite search budget.** Expensive analytical queries had to be bounded so that reporting never degraded the platform for everyone else.
- **Alert fatigue.** An empty "nothing happened" email trains a client to ignore the product, so silence had to be a deliberate outcome rather than a default.
- **Auditable sends.** Support had to be able to answer exactly what was sent, to whom, and whether it arrived.
- **Non-production safety.** The same pipeline had to be exercised in staging without ever emailing a real client.
- **Shared delivery, separate promises.** Two report types of different urgency had to share one delivery mechanism without either being able to block the other.

## The Design

Having identified the assets and the threats to them, it is now possible to consider the safeguards. The pipeline is a queue-backed chain of small stages, arranged as a security wheel of detection, avoidance, protection, and recovery, that branches by report type and converges for delivery. Detection is provided by per-recipient send records, month-based email logs, and delivery callbacks; avoidance by a 15-minute ingestion buffer, empty-send suppression, and validation of a report's queries before the expensive work begins; protection by isolating the two report types on separate generation queues, bounding concurrency against the search budget, and persisting a durable artifact before any email leaves; and recovery by making every stage replayable from that artifact and reconciling every callback to the run that produced it. These safeguards are refined continually rather than fixed once.

Branching happens once, at the front. The report processor creates a run request, attaches a run identifier to the message, and routes it by type: real-time schedules go to a dedicated real-time generation queue, while program-insights schedules use the standard one. Both generation stages run with deliberately bounded concurrency, which keeps the expensive search work in check and makes any failure easy to isolate by run. From the artifact onwards the two promises share the same machinery.

Real-time scheduling is deliberately windowed, and this is where avoidance is exercised most directly. Each run is aligned to a regular interval and its query window is shifted back by a short ingestion buffer, so that the most recent minutes are left untouched and posts that arrive late are still captured on the following run. The generator gathers the report's topics and influencers, asks the search surface for matching posts, and records the activity it finds. If there are no matching posts and the run is not a preview, the send stage skips the email entirely, so an alert is never sent to say that nothing happened.

Program insights run on a weekly cadence. The scheduler waits until a full week has passed before the next run, then compares the current week against the previous one across the programme's overview, its trending topics, and its most-engaged influencers. Before committing to the full request, the generator validates the report's queries with a small search call, so that a badly shaped programme fails cleanly rather than producing a broken template. Every search call is wrapped in a Polly retry proxy that separates a transient failure, which is retried with backoff, from a permanent one, which stops at once, because a temporary outage and an invalid programme call for entirely different responses.

Both report paths produce a durable artifact before any email is sent, and this is the foundation of recovery. The generation stage writes the report data to S3, then queues the send stage. That stage reads the artifact, asks SES to render the template, saves the rendered HTML back to S3 so any historical email can be inspected exactly as the client received it, and sends the templated email. The per-recipient results are written to SQL Server so that later callbacks can be reconciled to the run that produced them, and in staging-like environments an override recipient reroutes all mail so the pipeline can be exercised without reaching a real client.

Closing the loop is where detection lives. Delivery, bounce, and complaint events arrive through SNS and are processed from queues. The callback processor updates the run status and indexes an email-log entry into Elasticsearch, while a separate processor consumes the same events through its own queue and forwards engagement to customer success, so that post-send tracking never couples to the send path. Previews exercise the whole machinery from end to end: a preview marks its schedule accordingly, applies the preview window rules, and feeds the ordinary report-processor queue, so what a client approves is exactly what the system produces in production.

The result is a reporting platform shaped around its promises. It isolates the two report types where their urgencies differ and shares the delivery machinery where they do not, spends the search budget only within bounded and validated windows, suppresses the empty alert that would erode trust, and preserves a durable artifact and a reconciled record for every send, so that the question a client or a support engineer might ask, whether a report was sent, what it contained, and whether it was opened, is always answered by the system rather than by guesswork.
