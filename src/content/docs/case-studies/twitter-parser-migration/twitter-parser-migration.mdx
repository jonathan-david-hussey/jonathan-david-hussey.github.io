## Context

The platform had years of downstream code built around one established shape for a tweet. Search indexing, reporting, enrichment, and ingestion all knew how to read that document. When the provider released a new major version of its API, it changed the contract: a tweet's data now arrived spread across related collections rather than as a single self-contained record.

The following assets have been identified as central to that dependency: the established tweet shape itself, the many downstream applications built upon it, the fidelity of each converted tweet, the derived display range that governs how a tweet is shown and indexed, and the compatibility of the library across every consumer's runtime. We are able to regard these as assets because a great deal of working software rested upon them; any negative impact, whether a broken shape, a mis-derived field, or a package that would not load in an older project, would affect not one tweet but every system downstream of it.

This framing turns the migration into a question of which dependencies must be protected, and by which safeguards, rather than a single task of reading the new format.

## The Problem

The threat was not merely renamed fields. A single response from the new API carries many tweets, each depending on users, media, places, polls, and referenced tweets scattered across shared collections, while the platform needed one coherent, self-contained document per tweet in the shape it already understood. Hardest of all, the value that records which part of a tweet should actually be displayed was removed or altered across the newer formats, yet display and indexing still depended upon it, so it had to be derived rather than copied.

The level of risk followed from likelihood and severity together. Forcing ten dependent applications through a coordinated, simultaneous migration carried a high likelihood of breaking something, and a conversion that silently produced a wrong shape or a wrong display range would corrupt downstream data with no visible failure at all. Our analysis therefore favoured a compatibility layer: absorb the provider change in one place, keep emitting the established shape, and let each consumer adopt the newer data when it was ready.

## Constraints

- **A frozen downstream contract.** Many systems consumed the established tweet shape, and none of them could be forced to migrate on the library's schedule.
- **Real-world variety.** Emoji-only tweets, surrogate pairs, nested quotes of retweets, and language-specific reply formats all had to be handled, not just the common case.
- **Many runtimes, many versions.** The library had to load and run in every downstream project, old and new, which meant targeting the broadest common runtime.
- **A derived, not copied, display range.** The range that governs display and indexing had to be reconstructed from the tweet itself, reliably enough to trust across the whole corpus.

## The Design

Having identified the assets and the threats to them, the safeguard is a single compatibility layer packaged for the broadest possible reach. The library takes a response from the newer API and returns the established shape, so the provider change is absorbed in one place. It is shipped as one NuGet package targeting .NET Standard 2.0, and that is the decision that matters most for compatibility: a single build runs in every downstream project regardless of its .NET version, so no consumer had to change runtime, or indeed change at all, in order to keep working. The remaining safeguards fall into a familiar shape: prevent the breaking change from reaching consumers, avoid fabricating data the newer format no longer provides, and detect drift early through fixtures.

A response arrives carrying many tweets over shared collections, so the first step is to make each tweet self-contained. The library builds one document per tweet and attaches the referenced tweets, users, media, places, and polls relevant to that tweet, so the converter can work on a single tweet at a time without losing the context a quote, retweet, or reply needs. A response that cannot be converted safely is skipped rather than allowed to corrupt the output.

Conversion itself is treated as a contract. Focused converters map each part into the established shape, rebuilding the tweet, the author, and the entities such as hashtags, mentions, links, and media. Values with no equivalent in the older shape are set to safe defaults rather than guessed, and values that exist only in the newer data are preserved to one side in a dedicated extension, so a consumer receives an honest older document and can reach for the new fields when it chooses to. Quotes and retweets are recursive: a referenced tweet passes back through the same converter, with a depth limit so that real-world nesting cannot run away.

Deriving the display range is the part that carried the most risk, because it had to be reconstructed rather than read. The rules came from machine-learning analysis over millions of tweets sampled across two years, and were then encoded as explicit, inspectable detectors instead of a model call. The library establishes whether a tweet is a normal post, a reply, a quote, or a retweet, extracts features from the text, such as whether it opens with a handle, whether it ends with a link, whether it carries media, and its length measured with awareness of surrogate pairs, and then runs an ordered chain of detectors, because replies are where the range is most ambiguous. A catch-all detector sits last so that even an unusual tweet still produces a bounded range. Because the rules are inspectable, a wrong result is debugged by its features and its matching fixture, not by interrogating an opaque model.

The test suite is built where the risk actually lives. Hundreds of real-tweet fixtures, most of them exercising display-range derivation, compare expected output against converted input for replies, retweets, quotes, normal posts, and nested references, and the same fixtures can be run over folders of sampled tweets to catch drift when a new tweet shape appears.

The result is a compatibility bridge shaped around the software that depends on it. It absorbs a breaking provider change in one place, emits the shape that many applications already understood, preserves the new-only data to one side, derives the display range from inspectable rules with fixtures around the hard cases, and ships as a single broadly targeted package, so that every consumer, whatever its .NET version, keeps working and migrates on its own schedule rather than on the provider's.
