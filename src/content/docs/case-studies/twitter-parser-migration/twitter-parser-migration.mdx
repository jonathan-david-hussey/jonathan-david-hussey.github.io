## Context

The platform had years of downstream code built around Twitter's v1 `Status` JSON shape. Search indexing, reporting, enrichment and ingestion workflows all knew how to read that document. Twitter v2 changed the contract: tweet data arrived as `data`, `includes`, referenced tweets, users, media, polls, places and errors spread across related collections.

## The Problem

The migration problem was not just "rename fields." A v2 timeline response contains many tweets, each depending on different users, media, places, polls and referenced tweets scattered across shared collections — while the old platform needed one coherent v1-style document per tweet, with enough nested structure preserved that downstream systems did not have to learn v2 all at once. Hardest of all, `display_text_range` was removed or changed across API formats, but display and indexing behaviour still depended on it: the field had to be *derived*, not copied.

## Constraints

- **A frozen contract.** Downstream systems needed v1-shaped output; none of them could be forced to migrate on the parser's schedule.
- **Real-world tweet weirdness.** Emoji-only tweets, surrogate pairs, nested quotes-of-retweets, language-specific reply formats — the rules had to survive all of it.
- **Old runtimes.** The library had to run everywhere the platform did, which meant targeting .NET Standard 2.0.

## The Design

The package was built as a `.NET Standard 2.0` library with a NuGet package definition. It exposed a small service surface for converting one v2 tweet response or a v2 tweets response into v1-style JSON, while keeping the heavier conversion logic inside domain services.

```text
Twitter v2 timeline / lookup response
  -> split into per-tweet v2 responses
  -> attach relevant includes and errors
  -> detect tweet type
  -> extract tweet features
  -> derive display_text_range
  -> convert into v1 Status JSON
```

### Splitting Responses

The first boundary was response extraction. A Twitter v2 timeline response includes shared `includes` collections, but downstream conversion is easier when each tweet has its own response document. The tweet service builds a dictionary keyed by tweet ID and, for each tweet, attaches the tweet itself, referenced tweets, users, media, places, polls and matching errors relevant to that tweet. That step let the converter operate on one tweet at a time without losing the context needed for quotes, retweets, replies and attachments.

### Conversion as a Contract

The converter maps v2 data into the older `Status` model through focused field converters. Top-level fields handle IDs, timestamps, language, text, source and public metrics. User conversion rebuilds screen name, profile counts, profile URLs, withheld countries and user entities. Entity converters rebuild hashtags, symbols, mentions, URLs, media and polls. Geo and place converters restore location-shaped data where v2 provides it.

Some v1 fields no longer existed in v2, so the converter deliberately set them to `null` or safe defaults instead of guessing. Some v2 fields had no v1 home, so they were preserved under a dedicated `V2` extension object: impression count, bookmark count and conversation ID.

Quotes and retweets were recursive: a referenced v2 tweet passes back through the same converter, with a maximum recursion depth to avoid runaway nesting on real timeline data.

### Display Range Rules

The rules for deriving `display_text_range` came from machine-learning analysis over millions of tweets sampled across a two-year window, then encoded as explicit feature-based detectors. The code first detects whether the response is a normal tweet, reply, quote or retweet, then extracts features from the text:

```text
starts with handle
reply handle at start or end
has media
has URL
ends with external URL
tweet references itself
consecutive handles
consecutive URLs
language code
surrogate-pair-aware text length
```

Replies use an ordered detector chain because replies are where display ranges are most ambiguous — combinations such as "starts with handle and ends with URL," "does not start with handle but has media," and language-specific exceptions. A catch-all detector remains last in the chain so unusual tweet shapes still produce a bounded range.

### Testing the Edge Cases

The fixture suite reflects the migration risk: 298 JSON fixtures, including 227 display-text-range fixtures and 51 v2-to-v1 converter fixtures. Converter tests compare expected v1 JSON against converted v2 JSON for replies, retweets, quotes, normal tweets and nested references. Display-range tests cover emoji-only tweets, non-ASCII text, many handles, many URLs, media URLs, external links, self-references and language-code exceptions. Integration tests can run over folders of sampled tweets to catch parser drift when new tweet shapes appear.
