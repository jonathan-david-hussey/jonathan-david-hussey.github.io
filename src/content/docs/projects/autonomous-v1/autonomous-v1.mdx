## Context

Autonomous Vehicle Backend is a coordination system for a small fleet-style operating model. The assets under control are vehicles, missions, assignments, telemetry records, and mission status evidence. Each one has a different operational value: vehicles identify the field device, missions describe the intended work, assignments bind that work to a vehicle, telemetry reports where the vehicle is and what state it is in, and status entries explain how a mission progressed.

The project is built as a NestJS backend with MongoDB persistence and EMQX-based MQTT messaging. It exposes a conventional API for operators, listens to vehicle-originated MQTT traffic, publishes mission instructions back to vehicle-specific topics, and provides JSON mission reports that combine the relevant records into an operational view.

The repository also includes the supporting environment needed to demonstrate the system. Docker Compose starts the MQTT broker, MongoDB, the API, and a Python telemetry sender; Mongo initialization seeds a sample vehicle and mission; Swagger documents the HTTP surface; and the Postman collection gives reviewers a direct route through the main flows.

## The Problem

Autonomous vehicle coordination is not a single request-response problem. An operator can create a mission through an API, but a vehicle reports its telemetry asynchronously, may receive mission instructions through a broker, and returns mission progress later as a separate message. If these communications are treated as one flat data model, it becomes difficult to tell which asset was affected, which channel carried the update, and which transition actually occurred.

The backend therefore had to manage two related risks. The first was operational ambiguity: a mission assignment should not vanish into a final status without showing when it was queued, when it started, and how it ended. The second was integration fragility: telemetry and mission traffic should be simple enough to exercise locally, but structured enough that later safeguards such as broker ACLs, TLS, retry policy, and observability have obvious places to attach.

## Constraints

- **Mixed communication styles.** Operator commands and reports need HTTP semantics, while vehicle telemetry and mission status updates fit MQTT topic traffic.
- **Registered-vehicle validation.** Telemetry should not be persisted when the vehicle identifier is missing or unknown, because unregistered messages would weaken the integrity of the operational record.
- **Lifecycle visibility.** Mission assignment needs a timestamped status history rather than one mutable field, so reports can reconstruct the sequence of events.
- **Local demonstration.** The system had to run without managed infrastructure, which meant the broker, database, API, simulator, seed data, and sample credentials all needed a Dockerized path.
- **Transport boundary.** Broker topic patterns should not leak into every service. MQTT concerns need to be translated into typed application events before domain processing.
- **Production hardening path.** The implementation had to leave room for broker ACLs, TLS on MQTT and MongoDB, structured logging, retries, rate limiting, health checks, metrics, and stronger persistence choices.

The diagram deliberately reduces the system to the main operational path. The HTTP API captures administrative intent, the event handoff turns a mission assignment into broker traffic, the vehicle or simulator responds with telemetry and status messages, and MongoDB keeps the evidence that later feeds reports.

At application startup, NestJS connects an MQTT microservice to EMQX using MQTT v5, a generated backend listener client id, and QoS 1 subscription options. The same bootstrap configures URI API versioning, a global validation pipe that strips unknown fields, a global exception filter, JWT-guarded controllers, and Swagger at `/api`.

### HTTP Surface

The protected API is intentionally small and operational:

- `POST /vehicle`, `GET /vehicle`, `GET /vehicle/:id`, and `PATCH /vehicle/:id` manage vehicle records.
- `POST /v1/mission`, `GET /v1/mission`, and `GET /v1/mission/:id` manage mission definitions.
- `POST /v1/vehicle-mission`, `GET /v1/vehicle-mission`, and `GET /v1/vehicle-mission/:id` assign missions to vehicles and expose assignments.
- `GET /v1/telemetry/vehicle/:id` returns telemetry for a registered vehicle.
- `GET /v1/report/missions` composes the mission report from assignments, vehicles, missions, and status histories.

The controllers do not try to simulate the vehicle channel. They validate and persist intent, then emit internal events where the rest of the operational flow can continue.

### MQTT And Event Boundary

The MQTT controller listens to three topic families:

1. `vehicle/+/telemetry`
2. `vehicle/+/mission`
3. `vehicle/+/mission/status`

Each handler performs a simple translation. It receives a typed payload and emits an application event such as `vehicle.telemetry`, `vehicle.mission`, or `vehicle.mission.status`. That event service is deliberately narrow: it wraps Nest's event emitter and preserves a typed mapping between event topics and payload shapes.

The listener layer then owns the side effects. Telemetry events call the telemetry service, which rejects messages with no vehicle id and attempts to find the vehicle before writing the telemetry record. Mission-status events create a new status entry for the supplied vehicle-mission id. Create-vehicle-mission events update the assignment to `QUEUED` and publish the mission payload to `vehicle/{vehicleId}/mission`.

### Mission Lifecycle

Mission assignment is the central workflow. The HTTP request first verifies that both the vehicle and mission exist. The service then creates the vehicle-mission assignment and immediately records `PENDING`. The controller emits `create.vehicle.mission`, and the create listener records `QUEUED` before publishing to the vehicle mission topic.

For the local demonstration, a vehicle mission listener simulates the vehicle side of the exchange. When a mission message appears, it publishes `IN_PROGRESS`, waits briefly to represent execution, then publishes either `COMPLETED` or `FAILED`. In a production deployment, that simulation boundary would be replaced by real vehicle firmware or an integration service, while the backend contract would remain the same.

The important safeguard is that no lifecycle transition is overwritten. Statuses are append-only records with their own ids, mission-assignment id, status value, and timestamp. This makes reporting more useful than a final-state lookup because it can explain how the system reached that state.

### Persistence And Reporting

MongoDB stores five operational collections through TypeORM entities: `vehicles`, `missions`, `vehicle_missions`, `vehicle_mission_status`, and `telemetry`. The entities use generated UUID-style public ids, Mongo object ids internally, and indexes on identifiers that are used for lookups and joins. Telemetry records carry the vehicle id, timestamp, GeoJSON-style point location, and optional status.

The report service treats the database as an evidence store. It loads all vehicle-mission assignments, gathers the distinct vehicle and mission ids, loads related statuses, vehicles, and missions, builds lookup maps, and returns each assignment with the vehicle, mission, and status sequence attached. That shape is more valuable for operators than a flat mission list because it shows both responsibility and progression.

### Local Runtime

The Docker Compose stack makes the integration path visible:

- `emqx` exposes MQTT and the dashboard ports for the broker.
- `mongodb` initializes the database, app user, seed vehicle, and seed mission.
- `autonomous_api` runs migrations, starts the NestJS API, connects to MongoDB, and connects to EMQX.
- `telemetry_sender` packages a Python MQTT client that can publish location and status messages to a selected vehicle topic.

This is not the final production posture. The improvement backlog correctly identifies the next safeguards: TLS, broker ACLs, least-privilege container users, stronger JWT signing, structured logging, OpenTelemetry, Prometheus/Grafana metrics, retries and backoff, dead-letter handling, rate limiting, health checks, and load testing. The value of the current design is that these concerns already have clear boundaries: broker security belongs at EMQX and MQTT client configuration, data integrity belongs in services and repositories, and operational visibility belongs in events, status history, and reports.
