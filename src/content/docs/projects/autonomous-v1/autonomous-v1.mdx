## Context

The Autonomous Vehicle Backend is a NestJS system for coordinating autonomous vehicles through a mix of HTTP APIs, MQTT messaging, and MongoDB persistence. It manages vehicle records, mission definitions, vehicle-mission assignments, telemetry intake, mission status history, and JSON mission reports.

## The Problem

Autonomous vehicle coordination needs both request/response management and asynchronous field communication. The backend had to expose a conventional API for operators while also accepting telemetry, dispatching missions, and recording mission status updates over MQTT.

## Constraints

- **Mixed communication styles.** HTTP fits operator commands and reports; MQTT fits vehicle telemetry and status traffic.
- **Local demonstration.** The stack needed to run without a hosted broker or database.
- **Validation before persistence.** Telemetry should only be stored for registered vehicles.
- **Lifecycle visibility.** Mission assignment needed a history, not just a final state.
- **Production hardening path.** The design had to leave room for broker ACLs, TLS, observability, retries, rate limiting, and stronger persistence/indexing choices.

## The Design

The runtime stack is designed to be started locally with Docker Compose. It brings up an EMQX MQTT broker, MongoDB, the NestJS API, and a Python telemetry sender that can publish simulated vehicle telemetry to the broker. Swagger is exposed at `/api`, and the HTTP API uses URI versioning, JWT guards, global validation, and a global exception filter.

The MQTT side listens to three topic patterns:

1. `vehicle/+/telemetry`
2. `vehicle/+/mission`
3. `vehicle/+/mission/status`

Incoming MQTT messages are routed through NestJS message handlers into an internal event service, which decouples broker concerns from domain processing. Telemetry messages are validated against registered vehicles before they are written to MongoDB. Mission status responses are persisted as timestamped status records.

### Mission Assignment Flow

Mission assignment is an event-driven flow. The HTTP API creates the vehicle-mission record, stores an initial `PENDING` status, emits a create-mission event, then a listener records the `QUEUED` status and publishes the mission payload to the target vehicle topic at `vehicle/{vehicleId}/mission`. A separate listener handles status responses such as `IN_PROGRESS`, `COMPLETED`, and `FAILED`.

Persistence is handled through TypeORM over MongoDB. The data model includes vehicles, missions, vehicle-mission assignments, mission status entries, and telemetry records. Telemetry entries include a generated telemetry id, vehicle id, timestamp, GeoJSON-style location, and optional status.

### Reporting And Hardening

The reporting path composes mission reports from multiple repositories. It joins vehicle-mission assignments with their vehicles, missions, and status histories so the API can return an operational view of what was assigned, to which vehicle, and how the mission progressed.

The project also includes a Postman collection, migration tooling with `migrate-mongo`, Jest unit and e2e tests, and an improvement backlog covering production hardening areas such as broker ACLs, TLS, observability, retries, rate limiting, and stronger persistence/indexing choices.
