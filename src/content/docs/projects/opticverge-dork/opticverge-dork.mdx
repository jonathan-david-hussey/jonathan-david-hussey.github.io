## Context

Opticverge.Dork started as a CLI for packaging repository context for LLMs, then grew into a local knowledge orchestration platform for personal document collections. The CLI scans a codebase, enriches files with Git metadata, filters by contributor and extension, strips noise such as C# single-line comments, converts HTML to Markdown, and writes a single Markdown context file for downstream model work.

## The Problem

The first CLI solved one narrow task: make a codebase easier to hand to an LLM. The broader need was to ingest mixed local knowledge sources, parse them into searchable records, preserve citations, and plan study guides from retrieved evidence without losing the simplicity that made the CLI useful.

## Constraints

- **Local-first operation.** The platform needed to run as a developer-local topology, not depend on managed services for the first usable version.
- **Layer discipline.** Domain, application, infrastructure, API, worker, parser, CLI, and test concerns needed clear ownership.
- **Parser variability.** Rich parsing should be available when Python dependencies are present, with a fallback path when they are not.
- **Search evolution.** The design needed room for OpenSearch or Qdrant later while keeping PostgreSQL canonical now.
- **Maintainability gates.** Architecture tests and analyzer thresholds needed to prevent the system from collapsing into one large ingestion script.

## The Design

The platform layer takes the original CLI idea further: ingest, parse, index, search, and plan study guides over local knowledge sources. The solution follows Clean Architecture and Domain-Driven Design, with separate Domain, Application, Infrastructure, API, worker, parser, Aspire, service-defaults, CLI, and test projects.

The local Aspire topology runs an ASP.NET Core API, an ingestion worker, PostgreSQL with pgvector, Apache Tika, and a Dockerfile-built Python parser service. OpenSearch and Qdrant are modeled as opt-in future resources, while the v1 write path keeps PostgreSQL as the canonical relational, JSON metadata, full-text search, and vector store.

### Durable Ingestion

The ingestion worker claims durable jobs from PostgreSQL and processes them idempotently. Its six stages fingerprint the source, extract metadata, extract page text and artifacts, map pages into useful sections while suppressing low-value pages, generate tags and concepts, and persist canonical records, FTS rows, and embeddings.

The Python parser service exposes `/health` and `/parse`, uses `pypdf` when available, emits parse manifests, page artifacts, page classifications, sections, and keyword-style tags, and falls back to text extraction when a richer parser is unavailable. A .NET fallback parser client sits behind the same application abstraction.

### Search And Study Guides

Search is exposed through `/v1/search` and currently ranks PostgreSQL-backed chunk candidates with lexical rank, tag overlap, and section quality. The study-guide planner uses search results to build objective-oriented sections with query variants, page-selection rules, and candidate cited sources, then persists the plan for later retrieval.

The project also treats architecture and maintainability as first-class constraints. ArchUnitNET tests enforce layer isolation, infrastructure convention tests keep persistence ownership disciplined, and analyzer gates enforce thresholds for cyclomatic complexity, cognitive complexity, method size, file length, nesting depth, parameter count, and related maintainability metrics.
