## Context

DORK stands for Data ORchestration & Knowledge. The long-term intention is a toolset that can collect different forms of data and prepare them for consumption by large language models. In the present iteration, however, the completed asset is narrower and more concrete: a .NET command-line tool that traverses a repository and produces an LLM-ready Markdown context pack.

This distinction is important. The repository contains early platform work around Aspire, API services, ingestion workers, parser services, and search-oriented design. Those elements indicate the direction of travel, but the current article focuses on the part that has been taken far enough to describe as working software: the repository scanner and Git-aware packager.

The CLI is intended for a common development problem. A codebase contains the evidence needed to answer a technical question, but the relevant evidence is distributed across many files, generated folders, old commits, and contributor histories. If that material is handed to an LLM without selection, the model receives noise. If it is selected manually, the result depends on memory and guesswork. DORK is the first pass at making that selection more controlled.

## The Problem

The key to preparing repository data for model consumption is not simply to concatenate files. A useful context pack needs to identify which files are relevant, remove low-value areas of the tree, include enough provenance to explain why a file appears, and keep the final output readable by both a person and a model.

There are several vulnerabilities in an unmanaged repository export. Build folders, dependency folders, binary assets, screenshots, and generated outputs can dominate the pack while adding little evidence. Files that have no relationship to the contributor or change area under review can distract from the question. HTML can be awkward to read in raw form. Git history may contain the only practical signal that a file belongs to a particular developer, feature, or line of investigation.

DORK therefore treats the repository as the asset under assessment and Git history as one of the controls used to reduce irrelevant material. The current implementation does not attempt to solve all knowledge ingestion. It solves the first collection step: traverse the tree, apply explicit filters, read the relevant history, and write a coherent Markdown artifact.

## Constraints

- **Repository scale.** A useful scan has to traverse a whole tree while avoiding obvious noise such as `bin`, `obj`, `node_modules`, IDE folders, packages, and image extensions.
- **Contributor relevance.** The pack should be able to focus on files touched by one or more contributors rather than every file in the repository.
- **Git availability.** Metadata lookup should enrich the output when a repository can be discovered, but the CLI must continue to operate when Git metadata is disabled or unavailable.
- **Selection clarity.** Include globs, excluded directories, excluded extensions, ignore patterns, and merge-commit behavior need to be explicit command-line inputs.
- **Readable output.** The result should be one Markdown file that can be reviewed before being handed to an LLM.
- **Content cleanup.** The output needs light preparation, including stripping C#-style single-line comments and converting HTML or HTM files to Markdown.
- **Local operation.** The tool should work from a positional source path, a `--source-dir` flag, the `DORK_SOURCE_DIRECTORY` environment variable, or the current working directory.

The CLI starts by resolving its operating boundary. `DorkOptions` accepts either a positional source directory or `--source-dir`; if neither is supplied, it falls back to `DORK_SOURCE_DIRECTORY` and then the current working directory. The same options object records include globs, excluded directories, excluded extensions, ignored path substrings, merge-commit handling, Git metadata behavior, date sorting, output path, and maximum parallelism.

Once the boundary is declared, `SourceFileProcessor` enumerates the repository recursively. It begins from the include globs, then removes files whose paths contain excluded directory patterns, whose extensions match excluded extensions, or whose paths contain ignored substrings. The defaults are deliberately defensive for repository scans: build outputs, IDE metadata, test folders, package folders, dependency folders, and PNG assets are excluded unless the user changes the selection.

Git metadata is the next control. When enabled, the tool discovers the repository with LibGit2Sharp and queries the history for each selected file path. `GitMetadataReader` can include or exclude merge commits, then applies contributor filters by matching supplied names against commit author names. If a contributor filter is active and no relevant history remains for a file, that file is removed from the final pack. If relevant history exists, the oldest relevant commit in the filtered history supplies the date and author details used in the section header.

File processing runs in parallel through `Parallel.ForEachAsync`, with the default maximum parallelism set to twice the processor count. Each accepted file is represented as `FileProcessingInfo`: the file path, optional Git creation date, author list, contributor signatures, and a flag indicating whether the contributor filter was satisfied. This keeps traversal, metadata collection, filtering, and writing as separate stages rather than collapsing the whole process into one loop.

The output stage is intentionally plain. Before writing a file, the processor removes lines that begin with `//` after leading whitespace. If the file is HTML or HTM, ReverseMarkdown converts it to Markdown. Each file is then appended under a heading in the form:

```markdown
### filename - dd/MM/yyyy - Authors: author name

file contents
```

When date sorting is enabled, files are ordered by Git-derived creation date descending and then by filename. The final artifact is a single Markdown file, written either to the configured `--output` path or to `{source-directory-name}-output.md` in the scanned directory.

The larger repository still contains architecture for future knowledge orchestration: API endpoints, ingestion worker concepts, parser services, Aspire hosting, persistence, and search. Those areas are not the current claim of DORK. They are the longer-term action plan. The present safeguard is the CLI boundary: select repository evidence with explicit rules, attach Git provenance where possible, and produce a context pack that can be inspected before it is used by an LLM.
