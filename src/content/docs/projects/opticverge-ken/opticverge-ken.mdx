## Context

Ken is a book-to-bank system. It takes a source PDF, derives a reviewable source profile, generates a verified multiple-choice question bank, and delivers that bank through a cross-platform Flutter practice app.

The project started from a practical study problem: a learner does not need another folder of static PDFs. They need reliable practice material that can explain where an answer came from, return to weak areas, and keep enough local history to make repeated revision useful.

The current phase uses bundled JSON banks and rendered page assets. That keeps the app simple, offline-friendly, and testable while leaving the path open for a later API, ingestion worker, PostgreSQL with pgvector, and search services.

## The Problem

Generic quiz generation can produce volume without trust. A model can miss important concepts, repeat the same idea, drift away from the source, or produce an answer that sounds plausible but cannot be traced back to the PDF.

The central problem was therefore not "generate questions." It was to design a repeatable process where questions are created from detected source structure, checked against explicit knowledge points, and rejected when their evidence cannot be found in the source text.

The app had a related problem. If the generated bank was useful, the learner still needed more than a static quiz viewer. The practice surface had to support book, chapter, timed, weakest-first, and focus-mode workflows, while recording enough local history to make weak areas visible.

## Constraints

- **Publisher-specific layouts.** Different publishers use different table-of-contents, chapter, section, appendix, and page-number conventions, so the pipeline needs detector strategies rather than one universal layout rule.
- **OCR and source quality.** PDFs vary in text extraction quality; detected structure has to be reviewable instead of silently trusted.
- **Source grounding.** Each accepted question needs a citation, page reference, and source quote that can be located in the extracted text.
- **Coverage over volume.** A fixed question count is not enough; generation needs to chase uncovered knowledge points until the configured coverage target is met.
- **Resumable LLM work.** OpenRouter-backed stages are long-running and rate-limit-sensitive, so they need worker-based concurrency, disk caches, and safe resume points.
- **Per-book isolation.** Terms, generated questions, source profiles, page images, UUIDs, and caches must stay separated by book.
- **Bundled-bank delivery.** The current app phase consumes generated assets locally before the larger backend exists.

Ken is split around one durable boundary: the verified generated bank. The Python generation system owns PDF ingestion, source profiling, term extraction, LLM-backed generation, verification, coverage reporting, and asset publication. The Flutter app owns library navigation, practice modes, page-reference viewing, session state, and local learner analytics.

The pipeline starts with `build_book.py`. A book build derives the book id, bootstraps metadata, creates a source profile, generates chapters, renders page and cover assets, publishes the per-book bank, and updates the app library index. Each book writes to its own data tree and cache prefix so long runs can resume without mixing artifacts from another source.

The ingestion layer accepts that PDF structure is not uniform. It uses detector strategies for embedded outlines, publisher conventions, section-numbered tables of contents, OCR-heavy textbooks, and reviewed structure overrides. The result is a source profile that records detected chapters, sections, landmarks, warnings, page counts, and extraction status for review.

Term extraction combines unsupervised, library-backed signals rather than hand-written domain lists. The pipeline uses spaCy noun chunks and POS-aware tokens, scikit-learn TF-IDF, YAKE keywords, KeyBERT semantic relevance, wordfreq common-language scoring, spelling normalization, and fallback tokenization. The clustering pass embeds candidate terms with Sentence Transformers, groups near-duplicates by cosine similarity, chooses a canonical display term, and keeps the merged variants as aliases. Those terms become the vocabulary reviewed by the pipeline and the tags attached to questions in the assembled bank.

The clustering step turns overlapping phrases into one reviewable vocabulary: for example, variants such as `service discovery`, `service registry`, and `registry lookup` can become one canonical tag with aliases, while timing phrases such as `latency`, `response time`, and `round-trip time` stay linked rather than scattering across the bank.

Question generation is coverage-driven. The generator enumerates atomic knowledge points for each page, targets uncovered material, and loops until embedding-similarity coverage reaches the configured target. It does not ask for a fixed number of questions per chapter; the question count emerges from the source.

Verification is part of the generation contract. Each generated question must include a verbatim quote, and the verifier checks that the quote can be found in the source text. It also rejects meta-framing, non-English output, answer-length outliers, duplicate stems, duplicate options, and other failure modes before questions reach the bank.

The OpenRouter-backed stages were optimized for throughput and recovery after the initial implementation. Page-level generation, knowledge-point extraction, and term judgement use configurable worker pools. `run_chapter.py` can start question generation in the background while term judgement runs, and per-book LLM caches make reruns and resumed builds cheaper. The CPU-side performance work is measured separately: the spaCy candidate-parsing benchmark improved from 4.11 seconds to 0.87 seconds by batching parser work, a 4.7x improvement with identical output.

The Flutter app consumes the bank and profile assets as a local library. Learners can practice a whole book, drill a chapter, run timed sessions, revisit the weakest questions, or use focus mode to retry mistakes. During a session the app records answer changes, hint usage, reference opens, elapsed time, pause/resume state, and correctness through Drift-backed local storage.

## Flutter App

The Flutter app is the learner-facing half of Ken: a local study library, a chapter-aware practice launcher, and the quiz surface that keeps the source evidence close to each question.
