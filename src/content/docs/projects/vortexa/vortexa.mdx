## Context

Vortexa is a React and TypeScript dashboard built for a technical challenge. The application loads a public GeoJSON dataset of boat ramps, summarizes the data through charts, and displays the same feature collection on a Leaflet map. Its purpose is not to model a large platform; it is to show how quickly a small geospatial dataset can be turned into a coordinated dashboard.

The key assets in the application are the GeoJSON feature collection, the Redux slice that owns the original and filtered views, the derived selectors, and the three dashboard widgets. Each asset has a distinct role. The API boundary fetches the data, the store keeps the canonical state, selectors turn features into chart-ready values, and the UI makes those values visible and interactive.

The project was developed over a short challenge window. That constraint shaped the implementation: use mature libraries for the expensive parts, keep custom code focused on state and data flow, and add enough tests to make the core interactions visible without overbuilding the application.

## The Problem

The underlying data is spatial, categorical, and numeric at the same time. A list of boat-ramp records is not very useful by itself. A reviewer needs to understand where ramps are located, which construction materials appear most often, and how the ramp areas fall into useful size categories.

The problem was therefore not only to render a map or a chart. The application had to provide a shared view of the data where the map and charts could respond to the same filter state. If a user selects a construction material, the visible map and size chart should reflect that subset. If the map bounds change, the chart data should be recalculated from the features visible in the current viewport.

This required a state model that was small enough for a challenge app but explicit enough to carry the dashboard behavior.

## Constraints

- **Time-boxed delivery.** The project was implemented as a coding challenge, so library choice and scope control mattered.
- **Remote data source.** The app fetches a public GeoJSON file at runtime rather than bundling a local dataset.
- **Geospatial shape data.** Features are rendered as GeoJSON multipolygons and filtered by map bounds rather than simple point markers.
- **Coordinated views.** Chart selections and map movements need to update a shared feature collection, not only local component state.
- **Responsive layout.** The dashboard has to fit summary charts and a map into a simple Material UI grid.
- **Testable state.** The filtering behavior belongs in Redux reducers and selectors where it can be exercised separately from chart and map rendering.

The application starts in `App.tsx`. On first render it checks the Redux request status, dispatches `fetchBoatRampsAsync` when the state is `idle`, and renders the dashboard. The API layer is deliberately small: `getBoatRampsAsync` fetches the public GeoJSON file and returns a `FeatureCollection`.

The Redux slice is the center of the system. It stores `data`, which is the original feature collection, and `sharedView`, which is the feature collection currently visible to widgets. It also stores request status, an incrementing `version`, and optional filter state for material, size category, and map bounds.

The material chart uses a Nivo bar chart. Its selector first creates a sorted set of all materials from the original data so empty categories remain visible, then counts matching materials in the current shared view. Clicking a bar dispatches `updateMaterialFilter`, toggling that material and clearing the size and map filters.

The size chart uses a Nivo pie chart. Its selector groups features into three area bands: `0-50`, `50-200`, and `200-526`. Clicking a slice dispatches `updateCategoryFilter`, toggling the selected size category and rebuilding the shared view from the original feature collection.

The map uses React-Leaflet. `Dashboard` computes the initial map center from the first multipolygon feature and wraps `BoatRampsMapWidget` in a `MapContainer`. The map widget renders the filtered GeoJSON and listens for `zoomend` and `moveend`; each event dispatches the current north-east and south-west bounds so the store can filter features whose multipolygon bounds intersect the visible map area.

## Implementation Notes

The UI composition is intentionally direct. Material UI `Container`, `Grid`, and `Paper` components provide the dashboard frame. The chart widgets own only their library-specific rendering and click dispatch. The map widget owns Leaflet events and rendering. Derived data lives in selectors rather than in component-local transformations.

The `version` field is a practical bridge between Redux and React-Leaflet. Because the GeoJSON layer can be sticky when data changes, the widget uses a `map-${viewVersion}` key to force the layer to remount after a filter update. It is not an elaborate abstraction, but for the size and purpose of the challenge it keeps the map honest.

The project uses CRACO to support the `@app/*` path alias from `tsconfig.paths.json`, keeping imports short across features, stores, hooks, and API modules. Tests use React Testing Library for render coverage around the app, dashboard, and widgets, plus reducer tests for the core boat-ramp state transitions.

## Archival Status

It must be noted that Vortexa is a technical-challenge project rather than a maintained production dashboard. It reflects a short implementation period, React 17-era dependencies, Create React App conventions, and a narrow public dataset.

As a portfolio project, its value is in the data-flow design. It demonstrates how to take a GeoJSON source, make Redux the shared state boundary, derive chart data from selectors, use charts as filters, and keep a map synchronized with the same filtered feature collection.
