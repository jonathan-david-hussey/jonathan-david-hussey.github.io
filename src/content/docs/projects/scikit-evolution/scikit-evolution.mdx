## Context

scikit-evolution is an archival Python framework for evolutionary machine-learning model search. Its core activity was to treat estimator configuration as an asset that could be represented, evaluated, mutated, and recorded. The main assets in the system are the shell learner abstraction, the parameter vocabulary, the data helper, the optimizer population, the cross-validation score, and the experiment evidence produced by each run.

The project was written during a period when model selection across scikit-learn, TensorFlow `skflow`, lightning, and related libraries required a great deal of manual coordination. Each estimator exposed different constructor arguments, default behavior, constraints, scoring characteristics, and training costs. Comparing them fairly meant more than selecting a different class name; it required a repeatable way to construct, score, and mutate each candidate.

scikit-evolution turned that process into a search system. A candidate is a `ShellLearnerSolution`: a shell learner instance, its generated keyword arguments, and its fitness. The optimizer creates a population of these candidates, scores them through cross-validation, mutates their parameter values, replaces weaker configurations, and persists the result for later inspection.

## The Problem

Manual model selection is vulnerable to inconsistency. A practitioner may try a random forest, then an SVM, then a neural network, but each experiment may use different preprocessing, different validation settings, different parameter ranges, or no durable record of the failed configurations. This makes it difficult to determine whether the best model was genuinely best, or simply the one that received the most attention.

The required result was therefore a common contract for heterogeneous estimators. A regressor, classifier, or clusterer needed to be represented in the same way from the perspective of the optimizer. Its parameters had to be generated and mutated through known rules, its dependencies had to be respected before construction, and its score had to be produced through the same validation boundary as the rest of the population.

The project also had to answer a practical question: when an evolutionary search finishes, what evidence remains? Without persisted arguments, scores, learner names, dataset metadata, and analysis queries, the search would produce a temporary answer rather than a reusable experiment.

## Constraints

- **Estimator variety.** Linear models, trees, ensembles, SVMs, nearest-neighbour methods, neural networks, Gaussian processes, kernel ridge models, naive Bayes, TensorFlow/skflow wrappers, lightning wrappers, and clustering shells behave differently.
- **Parameter shape.** Search has to handle integers, floats, booleans, categories, arrays, maps, bounded values, unconstrained values, gaussian values, and dynamic arrays through one mutation vocabulary.
- **Parameter dependencies.** Some model settings constrain others, such as `oob_score` depending on `bootstrap`, so valid construction requires dependency-aware argument generation.
- **Evaluation cost.** Every candidate is scored through real estimator training and cross-validation, which makes runtime, timeouts, and failed configurations part of the system design.
- **Experiment traceability.** Results need to preserve learner name, arguments, score, dataset size, normalizer, scorer, folds, problem type, and search settings.
- **Archival dependency landscape.** The repository reflects the Python ML ecosystem and research-code style of 2016, including deprecated library APIs and experimental wrappers.

The system is organized around the shell learner. `ShellLearner` accepts an estimator constructor and a dictionary of typed parameters. On initialization it generates keyword arguments from the parameter objects, stores optional arguments such as `n_jobs`, and records dependency rules that must be applied before the estimator is built. Calling `get_learner` creates the concrete scikit-learn, skflow, or lightning estimator for evaluation.

`ShellLearnerSolution` is the optimizer-facing representation. It contains the shell learner, the current keyword arguments, the fitness, and the minimum and maximum fold scores. This representation is deliberately higher-level than a simple numeric chromosome. The search is not only moving numbers; it is moving model configuration.

Learner helpers define which shell classes are available for a problem. `RegressorHelper` and related helpers group estimators by family and by data-size suitability. The repository includes shell classes for ensembles, linear models, trees, support vector machines, nearest neighbours, neural networks, Gaussian processes, kernel ridge models, naive Bayes, lightning learners, TensorFlow/skflow learners, and a clustering shell. This allows a run to search a narrow custom pool, such as an MLP-only experiment, or a wider model-family pool.

The Artificial Immune System path owns the main lifecycle. `ArtificialImmuneSystemBase.navigate` creates the population, scores it, then repeats sorting, mutation, scoring, and replacement until the configured epoch count is reached. The concrete `ArtificialImmuneSystem` selects random shell learners from the helper pool, constructs `ShellLearnerSolution` objects, and scores each with `model_selection.cross_val_score` using the validation options supplied by the run.

Mutation is rank-aware. The search builds distributions for mutation probability, number of dimensions to change, and rate-of-change variance. Stronger and weaker solutions therefore receive different mutation pressure. A clone is produced by selecting parameters from the current keyword argument map, mutating the corresponding `Parameter` objects, applying dependency rules when needed, and rejecting configurations that are identical to the parent or already trained.

## Parameter Vocabulary

The parameter system is the part that makes cross-family search possible. `EParameter` defines 15 categories: integer, real, string, boolean, array, map, category, bounded, mixed, dynamic, gaussian, Dirichlet, constrained, unconstrained, and kernel. The concrete parameter classes then describe how those categories should be generated.

A `BoundedParameter` carries a minimum and maximum value. An `UnconstrainedBoundedParameter` uses a starting range while allowing later movement beyond the original upper bound. A `CategoricalParameter` selects from a fixed list of options. A `BooleanParameter` handles binary flags. A `DynamicArrayParameter` can generate and mutate variable-length arrays, which is useful for neural-network layer definitions. A `GaussianParameter` has a pivot and step size for gaussian movement.

Randomization and mutation sit behind the same vocabulary. `ParameterRandomiser` selects an initial value based on the parameter type, while `ParameterMutator` handles bounded movement, category replacement, boolean changes, and dynamic-array value, add, remove, and swap operations. This means that an optimizer can mutate `n_estimators`, `bootstrap`, `hidden_layer_sizes`, `kernel`, `learning_rate`, and solver choices without hand-writing a separate mutation rule for each estimator.

The shell layer also manages dependency-sensitive configuration. For example, ensemble shells can constrain `oob_score` according to the current value of `bootstrap`, and bagging shells can constrain `warm_start` according to `oob_score`. These relationships are important because a search system that frequently constructs invalid estimators wastes generations on avoidable failures.

## Experiment Evidence

Scoring closes the loop. A solution is evaluated by asking its shell for a concrete learner, reading inputs and outputs from the data helper, and passing the estimator through cross-validation with the configured scoring function and fold count. The result is stored as average fitness plus minimum and maximum fold scores.

Persistence is part of that score boundary. The optimizer records problem name, problem type, timestamp, fitness, learner arguments, input count, class count, learner name, population size, epoch count, generation, fold score bounds, normalizer, searcher name, data source, and validation options. If MySQL configuration is available, the result is inserted into the configured table; otherwise it is appended to a file. Separate SQL helpers can then query available input sizes, best evolved learners, and top configurations for a learner and scoring function.

The repository also preserves the surrounding experimental material: problem scripts for classification and regression, custom metrics such as mean absolute percentage error, normalizers for time, temperature, and rate-of-change workflows, run outputs, analysis images, SQL scripts, and unittest files covering mutation, randomization, bounds, classifiers, regressors, preprocessing, SQL, GPU paths, metrics, verification, and database connectivity.

## Archival Status

It must be noted that scikit-evolution is preserved as research code rather than presented as a current package. The repository reflects 2016-era APIs, optional dependencies such as skflow and lightning, local database configuration, deprecated dataset references, and experimental modules that are not all equally complete.

As a portfolio project, its value is in the architecture of the search system. It demonstrates how many heterogeneous machine-learning estimators can be decomposed into shell learners, typed parameters, candidate solutions, validation scores, and persisted evidence. The lasting pattern is the separation between representation, mutation, evaluation, and analysis.
