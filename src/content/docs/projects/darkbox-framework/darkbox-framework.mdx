## Context

DarkBoxFramework is an archival Python research framework for automated object recognition. Its core activity was to discover image-processing pipelines rather than manually prescribe them. The assets of the system can be identified as the transform library, the DNA representation, the optimizer population, the classifier scoring loop, and the experiment records produced from each run.

The project came from a practical computer-vision problem. A useful object detector often depends on a sequence of preprocessing and feature-extraction choices: grayscale conversion, resizing, edge detection, morphology, texture features, thresholding, normalization, and finally classifier training. In a manual workflow each of those choices has to be selected, parameterized, tested, and revised by hand.

DarkBoxFramework turns that workflow into a search problem. A candidate pipeline is encoded as a sequence of transform names and parameters, built into concrete transform objects, applied to a labelled image set, and scored through a scikit-learn classifier. The best candidates are retained, mutated, replaced, and recorded as evidence for the next generation.

## The Problem

The problem was not simply to classify images. It was to identify whether the feature-engineering process itself could be automated in a way that remained measurable. If the framework only trained a classifier, then the preprocessing path would still be an external assumption. If it only generated image filters, then it would not know whether those filters improved recognition.

The required result was therefore a closed experimental loop. The framework had to build a candidate pipeline, apply it consistently to every image, derive classifier inputs, evaluate the result through repeatable metrics, and then feed that score back into the optimizer. Each step had to be recorded because a search that cannot explain which configuration produced a score is not useful after the run has finished.

This created a second concern: heterogeneity. The operations under consideration were not equivalent. A Gaussian blur has a sigma parameter, a local binary pattern has radius and point settings, an edge detector may have thresholds, and several morphology operations have structuring-element sizes. The framework needed a common representation without pretending the underlying operations were the same.

## Constraints

- **Large search space.** Transform order, transform selection, parameter values, classifier choice, image size, and dataset composition all affected the final score.
- **Heterogeneous image operations.** Filters, edge detectors, feature descriptors, morphology operations, thresholding, rank filters, and exposure adjustments had to be selected through one optimizer-facing mechanism.
- **Task-based fitness.** A candidate was only valuable if it improved recognition, so scoring had to be tied to classifier behavior rather than to a visual preference.
- **Expensive evaluation.** Every candidate required transforming the training image set, flattening feature arrays, and running cross-validation metrics.
- **Experiment persistence.** Long exploratory runs needed text, database, HTTP, and serialized learner paths so results were not lost when the process ended.
- **Archival code status.** The repository reflects 2014 to 2016 research code, including hardcoded paths, stubs, Python 2 era conventions, and exposed local configuration values rather than a current production package.

The system is arranged around a durable boundary: the encoded pipeline. The `DNA` object stores a list of tuples, where each tuple identifies a transform and the parameter values required by that transform. That representation is the common form used by the optimizer, the transform builder, persistence, detection scripts, and exported learner filenames.

The transform registry is the system's catalogue of possible operations. `TransformBase` maps a transform name to a constructor and to its parameter boundaries. It includes operations such as Gaussian and median blur, Canny, Sobel, Scharr, Roberts, Prewitt, Harris and FAST corners, HOG, local binary patterns, Gabor filters, histogram and exposure operations, Otsu and thresholding, rank filters, dilation, erosion, opening, skeletonization, white tophat, and black tophat. The framework is therefore not searching over one feature type; it is searching over families of operations that affect the image in different ways.

The DNA is materialized by `ImageCell`. Given a DNA list and the transform registry, it retrieves each constructor, reads the transform structure, creates the transform object with the selected parameters, and returns the ordered transform sequence. This provides the link between an abstract candidate and a runnable image-processing pipeline.

Image handling is delegated to `ImageSetHelper`. It reads class-labelled image folders, builds positive and negative sets around the target label, converts images to grayscale, resizes them, and generates classifier outputs. When a candidate is evaluated, it applies the transform sequence to the processed image set with joblib parallelism, cleanses infinities and NaN values, rescales intensity, converts to `float32`, flattens the image arrays, and returns the classifier input matrix.

The main optimizer path is `AISSA`, a hybrid Artificial Immune System plus Simulated Annealing implementation. A run begins by creating a population of candidate DNA sequences within the configured minimum and maximum pipeline length. Each candidate is scored only when its fitness is unknown. Scoring uses stratified k-fold cross-validation and records accuracy, F1, precision, recall, and Matthews correlation before averaging the combined score vector into a fitness value.

Mutation operates at several levels. Parameter mutation uses Gaussian perturbation within the bounds declared by the transform registry. Structural mutation can swap transform order, add a newly generated transform, or remove an existing transform. The mutation probability is influenced by rank, so the optimizer can apply different pressure across the sorted population. Simulated annealing then changes the tolerance for accepting mutated candidates, allowing the run to occasionally move through a worse local state when the temperature permits it.

Persistence was treated as part of the experiment rather than an afterthought. Each scored solution stores the simulation id, fitness, DNA, target label, dataset counts, classifier name and configuration, population settings, generation, image dimensions, best and worst fold scores, and the full score vector. Depending on configuration, that evidence can be written to a MySQL table or to a timestamped flat file. Related scripts can export trained classifiers to `.pkl` files whose names carry the DNA and image size information, then load those learners later to run detection over images.

The surrounding optimizer folder also includes implementations or scaffolding for genetic algorithms, particle swarm optimization, ant colony optimization, artificial bee colony optimization, and genetic programming. Not all of those paths are as complete as `AISSA`; their presence reflects the research purpose of the repository. The framework was a place to test how different metaheuristic ideas could be made to act on the same encoded pipeline representation.

## Archival Status

It must be noted that DarkBoxFramework is preserved as research code rather than presented as a current library. The repository contains duplicated script logic, compatibility assumptions from the Python 2 to Python 3 transition, hardcoded local paths, database configuration files, and experimental stubs. Those issues are part of the historical record of the project, not deployment guidance.

As a portfolio project, its value is in the architecture of the experiment. It demonstrates how an object-recognition workflow can be decomposed into assets, how those assets can be represented for search, how candidate solutions can be evaluated against measurable classifier outcomes, and how results can be recorded so that a long-running exploration remains inspectable after the fact.
